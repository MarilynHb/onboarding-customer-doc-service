Part 1 · Customer service · Step 7

Docker

~2-3 days Containerize One command up

You already run MSSQL in Docker. Now you containerize the Api itself, so the whole service (api plus database) comes up with a single docker compose up. This is how we run things, and getting it working once makes the document service much smoother later.

What you'll build

  • A Dockerfile for CustomerService.Api using a multi-stage build (build stage, then a slim runtime stage).
  • An updated docker-compose.yml that runs both MSSQL and the Api, on the same network.
  • Correct configuration so that, inside the container, the Api reaches the database at mssql (the service name), not localhost.
  • A startup that applies migrations (either automatically on boot, or via a documented command).
  • A short README.md: how to start, stop, reset, and view logs.

What you'll learn

  • Dockerfiles and multi-stage builds (why the final image is small)
  • Docker Compose: services, ports, environment variables, networks, depends_on
  • Service discovery inside a compose network (why the hostname changes inside containers)
  • Persistent volumes, so your data survives a restart but can be wiped on demand
  • How to debug a containerized .NET app

Suggested approach

  1. Write the Dockerfile. Use the official multi-stage pattern from the Microsoft guide: a sdk image to build and publish, then a aspnet runtime image for the final, smaller container.
  2. Add the Api as a service in docker-compose.yml alongside MSSQL. Set depends_on so the Api waits for the database.
  3. The key gotcha: the connection string inside the container must point at mssql (the compose service name), not localhost. Pass it as an environment variable in compose. (See the Reference card's host-name table.)
  4. Decide how migrations get applied. The simplest learning approach: apply them automatically on startup in development. Note the tradeoffs (fine for dev, not for production).
  5. Run docker compose up. Hit your API with the .http file. It should behave exactly as it did running locally, just fully containerized now.
  6. Write the README so that anyone (including you in three weeks) can start the whole thing cold.
⌥
The hostname thing trips everyone up. Outside Docker, your Api talks to the database at localhost,1433. Inside the compose network, localhost means "this container," so it has to use the service name mssql instead. When something can't connect, this is the first thing to check. Write a note explaining why in your own words.
