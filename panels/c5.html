Customer Service · Step 5

The Service layer

~4 days Orchestration Onboarding flow

In Step 4 your handlers did the entity construction directly: take a request, build the customer graph, call ICustomerRepository.AddAsync, return a response. That was intentional — it made the wiring visible. It also made the handlers too long and too aware of business decisions that do not belong to them.

Once you have a flow — onboarding a customer is not one INSERT but an ordered set of steps that must succeed or fail together; approving an application is not a status change but a state transition with consequences elsewhere — that logic needs a home that is not the handler. This is the layer where those flows live.

Handlers ask "did this request arrive validly?" Services ask "what is the correct sequence of things to do to fulfil this request?"

What you'll build

Inside CustomerService.Service:

  • ICustomerOnboardingService and its implementation CustomerOnboardingService. The interface and its implementation both live in the Service project. The service is injected with ICustomerRepository from Domain — the same interface your handlers used in Step 4, now used one layer deeper.
  • An onboarding flow on this service that does the following in order, in a single transaction:
    1. Build a new Customer in Pending status with the contact and address fields from the request.
    2. Build the right profile based on CustomerType: IndividualProfile or CompanyProfile.
    3. Build the CustomerIdentification records.
    4. Build an OnboardingApplication in Draft status.
    5. Hand the whole entity graph to ICustomerRepository.AddAsync.
    If anything fails along the way, none of it should persist.
  • An approval flow: ApproveAsync(applicationId). Loads the application with its customer, checks the application is in a state where approval is allowed (InReview, not Draft or Approved), sets the application status to Approved, sets CompletedAtUtc, and flips the customer's Status from Pending to Active. Save once.
  • A Result or ServiceResult type in CustomerService.Common. A small type that wraps either a success value or a failure with reasons. The service returns one of these. The handler unwraps it and produces the response. This is how you communicate "this is not allowed right now" without throwing exceptions for control flow.
  • Updated handlers. The entity construction logic moves out of the handlers and into the service. Once you have done this, compare the handler before and after: 
    // Before (Step 4): handler builds the entity graph itself
public async Task<CustomerResponse> HandleAsync(CreateCustomerRequest request)
{
    await _validator.ValidateAndThrowAsync(request);
    var customer = new Customer { /* ... many lines ... */ };
    await _repository.AddAsync(customer);
    return MapToResponse(customer);
}

// After (Step 5): handler delegates to the service
public async Task<CustomerResponse> HandleAsync(CreateCustomerRequest request)
{
    await _validator.ValidateAndThrowAsync(request);
    var result = await _service.CreateAsync(request);
    return result.Match(ok => ok.Value, fail => throw new ...);
}
    The handler's job is now exactly: validate → call service → translate result. Nothing else.

What you'll learn

  • Orchestration vs. persistence: the service decides the sequence, the store performs the writes
  • Why a multi-entity operation should be one transaction, and how EF Core handles that with a single SaveChangesAsync
  • State machines: which transitions are allowed, what changes alongside a transition
  • Returning results instead of throwing exceptions for expected failures
  • Where business rules belong: not in handlers, not in entities (yet), but in this layer that owns the flow

Where to start

Start with ServiceResult in Common — it is small and everything else depends on it. Keep it minimal: a success case with an optional value, and a failure case with reasons.

Then define ICustomerOnboardingService in the Service project before implementing it. Think about what the handlers will need to call: at minimum, a method to create a customer and a method to approve an application. The interface signatures should reflect the handler's perspective, not the database's.

Implement CreateAsync first. This is where the entity construction logic from Step 4's handler moves. The service is injected with ICustomerRepository from Domain — the same interface, now one layer deeper. SaveChangesAsync inside the repository wraps all inserts in a single transaction; you do not need anything special for atomicity.

Then implement ApproveAsync. Think about what you need to load, what states are valid entry points for approval, what changes on both the application and the customer when approval succeeds, and what the service should return when approval is not allowed.

Once the service works, update the handlers. Inject ICustomerOnboardingService instead of ICustomerRepository. Register everything — service and its interface — in a new AddCustomerService extension method in the Service project.

⌥
Design note: why this isn't on the entity yet. A purist would put the approval rule on OnboardingApplication.Approve(), with a method that throws if called at the wrong time. That's a strong pattern and we will explore it. For now, the rule lives on the service. The reason: encapsulation as a topic is its own lesson, and we want this step to be about flow, not about where state-transition methods belong on an entity. You will revisit this trade-off later.
◆
C# depth moment: pattern matching on results. When the handler gets a Result back from the service, it has to handle both cases. Modern C# pattern matching makes this clean: return result switch { Success s => Ok(s.Value), Failure f => BadRequest(f.Reasons) };. Read up on pattern matching with switch expressions. It replaces a lot of if/else chains and the compiler tells you when you forgot a case.
