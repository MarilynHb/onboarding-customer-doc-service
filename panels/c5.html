Part 1 · Customer service · Step 5

The Store layer and the database

~4-5 days EF Core + MSSQL Docker enters

Time for a real database. You'll build CustomerService.Store with EF Core, run MSSQL in a Docker container, map your enum properly, and create your first migration. The big payoff: you swap the in-memory store for the real one in one line, and nothing else in your code changes.

What you'll build

  • A CustomerDbContext in the Store layer, with a DbSet<Customer>.
  • Entity configuration: map the Customer to a table, configure the OnboardingStatus enum (store it as a string, not a raw int, you'll see why), set max lengths, mark the email required.
  • An EfCustomerStore that implements ICustomerStore using the DbContext.
  • A docker-compose.yml running MSSQL locally.
  • Your first migration, applied to the database.

What you'll learn

  • EF Core: DbContext, DbSet, entity configuration, migrations, async queries
  • How to map an enum to the database (and why storing the string is safer than the int)
  • Running MSSQL in Docker (and the Apple Silicon caveat below)
  • Configuration: connection strings in appsettings and environment variables
  • The real payoff of the interface: swapping implementations without touching the service

Suggested approach

  1. Start with Docker. Get MSSQL running before you write any EF code:
services:
  mssql:
    image: mcr.microsoft.com/mssql/server:2022-latest
    environment:
      ACCEPT_EULA: "Y"
      MSSQL_SA_PASSWORD: "Your_strong_Pass123"
    ports:
      - "1433:1433"

If you're on an Apple Silicon Mac, the line above won't run. Use mcr.microsoft.com/azure-sql-edge instead. Run docker compose up and connect with Azure Data Studio to confirm it works before going further.

  1. Add the EF Core NuGet packages to the Store project (Microsoft.EntityFrameworkCore.SqlServer and Microsoft.EntityFrameworkCore.Design).
  2. Create CustomerDbContext with a DbSet<Customer>. Add entity configuration. For the enum, use .HasConversion<string>() so the database stores "Registered" not "0". (Write a note on why that's safer.)
  3. Create your first migration and apply it:
dotnet ef migrations add InitialCreate \
  --project CustomerService.Store \
  --startup-project CustomerService.Api

dotnet ef database update \
  --project CustomerService.Store \
  --startup-project CustomerService.Api
  1. Connect to the database and confirm the Customers table exists with the right columns.
  2. Write EfCustomerStore implementing ICustomerStore with real EF queries (ToListAsync, FindAsync, Add + SaveChangesAsync).
  3. In your DI setup, swap InMemoryCustomerStore for EfCustomerStore. Your CustomerService does not change at all. That's the lesson.
◆
C# depth moment: async all the way down. EF Core's database calls are genuinely async (real I/O now, unlike the in-memory fake). This is where await earns its keep. Read the difference between ToList() and ToListAsync(), and write a note: what is the thread doing while it awaits a database call?
