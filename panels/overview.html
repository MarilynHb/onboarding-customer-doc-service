Welcome

Your customer document service

An ~8-week guided build · Clean Architecture · microservices · Docker

Over the next ~8 weeks, you're going to build something real: a small but production-shaped microservices system inspired by what we actually do at Moneygate. By the end you'll have a working API for managing customer documents, a second service that consumes it, a Blazor UI on top, and everything orchestrated with Docker Compose.

More importantly, you'll have built it with proper Clean Architecture so the patterns translate directly to our real codebase.

★
This isn't a tutorial. It's a guided build. The point is not to finish, it's to understand every choice you make and be able to defend it.

Why this project

You've worked with .NET and APIs before. So a lot of what you'll see here might feel familiar at first. That's a trap. The work this time isn't how to write a controller, it's why we write it the way we do.

Three things make this project genuinely worth your time:

  1. It mirrors our real architecture. Clean Architecture, microservices that own their data, service-to-service communication over HTTP, Azure Blob for files. These are the patterns you'll see every day at Moneygate.
  2. It teaches by enforcing the architecture. You can't skip the Domain layer or shortcut the Application layer because the next milestone won't work without them. The structure forces depth.
  3. The features are EMI-flavored. Documents, customer onboarding, status transitions, audit trails. The domain you're modeling is the kind of thing we actually deal with.

What you're building

A system with three runnable applications that work together.

Documents.Api

The document service. Owns documents (both the files and the metadata about them) and exposes a REST API. Files are stored in Azure Blob (via Azurite locally), metadata in its own MSSQL database. This is where you apply Clean Architecture in full depth. No other service touches the documents database directly.

CustomerOnboarding.Api

Represents a banking app's onboarding flow. Owns customer records and onboarding applications. When a customer needs to upload an identity document, this service doesn't store the file itself. It calls Documents.Api to do that, and keeps only a reference. Its own MSSQL database tracks customers and application status.

CustomerOnboarding.Ui

A small Blazor Server frontend. Lets a customer register, start an onboarding application, upload documents, and check status. Admin pages for reviewing pending applications.

Everything runs locally in Docker Compose: both APIs, the Blazor UI, Azurite for blob storage, and the MSSQL databases. One command brings the whole stack up.

What you're learning

The technical surface you'll cover over these weeks:

  • C# in depth: records, generics, LINQ, async/await, pattern matching, delegates, value types
  • .NET ecosystem: the difference between C# the language and .NET the platform, the SDK, the runtime
  • Clean Architecture: Domain, Application, Infrastructure, Presentation layers and the dependency rule
  • REST API design: verbs, status codes, idempotency, the request pipeline
  • ASP.NET Core: controllers, middleware, dependency injection, configuration
  • EF Core: DbContext, migrations, async queries, change tracking
  • Azure Blob Storage: via Azurite, the Microsoft-provided local emulator
  • Service-to-service communication: IHttpClientFactory, typed clients, why HttpClient lifetime matters
  • Docker and Docker Compose: containerizing .NET apps, orchestrating a multi-service stack
  • Blazor Server: components, rendering, calling APIs from C# in the browser
  • Testing: unit tests with xUnit, mocking with Moq, integration tests with WebApplicationFactory

That's a lot. You won't master all of it. You'll get a solid foundation in each, and the project gives you the context to deepen any of them later.

How we'll work together

Each working day, post a short message in Teams at end of day. Three lines is enough: what you did, what you're stuck on, what's next. This keeps me in the loop without needing a meeting.

Real conversation at each milestone

When you finish a milestone (or near the end of one), we'll have a proper 30-45 minute call. We'll walk through your code, you'll explain your choices, I'll push back on the ones I disagree with, and we'll set up the next milestone together. That's roughly 8-9 conversations across the whole project.

Stuck threshold

If you've been stuck on something for more than an hour and you've tried at least two approaches, ping me. Stuck does not mean "I don't know the answer." Stuck means "I've tried, and I'm not sure where I'm wrong." That's exactly when to ask, and it's not a sign of weakness. The opposite. Pretending to be unstuck wastes everyone's time.

Pace is yours

The milestones are sized roughly in days, but I'm not measuring you against the calendar. If a milestone takes longer because you're going deep, that's a good thing. If it takes shorter because you flew through it, we'll talk about whether the depth was actually there. The goal is comprehension, not speed.

The journey at a glance

The project is organized in three parts. Each part builds on the last.

Part 1 — Documents Service

You build a complete Clean Architecture service from scratch. Domain, Application, Infrastructure, Web layer, with file storage via Azurite. By the end you have a working REST API for documents.

  • Milestone 1 — Orientation and foundations (~3 days)
  • Milestone 2 — The Domain layer with real depth (~5 days)
  • Milestone 3 — The Application layer and use cases (~5 days)
  • Milestone 4 — Infrastructure: EF Core, MSSQL, Azurite (~6 days)
  • Milestone 5 — Web layer, fake auth, first end-to-end slice (~5 days)

Part 2 — Customer Onboarding

You build a second service that consumes the first. This is where microservices stops being a buzzword. Then you add the Blazor UI on top.

  • Milestone 6 — The Onboarding service with HTTP integration (~6 days)
  • Milestone 8 — Blazor UI and stretch goals (~6 days)

Part 3 — Bringing it together

You polish the entire stack and orchestrate it with Docker Compose.

  • Milestone 7 — Full Docker Compose orchestration (~3 days)

I'd suggest reading every section once before starting. Get the shape of the journey, then come back to each as you reach it.

Things to keep in mind throughout

You will make mistakes

That is the entire point. Your first attempt at Clean Architecture will not be perfect. You'll put logic in the wrong place. You'll refactor on day three. You'll discover on day twelve that you should have done something differently on day eight. All of this is the work, not a failure. The intern who breezes through never learns as much as the intern who hits walls and works through them.

Read more than you write, early on

When you hit something new, look at how the Jason Taylor template does it. Look at how Microsoft's official samples do it. Most of your questions are already answered in well-known places. The skill isn't inventing solutions, it's recognizing patterns. You can also check our current repository to get inspired.

Defend your choices

By the end of this project, every line of code should be something you chose deliberately, even if "deliberately" means "I picked the simplest option that worked." Pretending you understand when you don't will catch up with you fast.

Push back when you disagree

If something in this document feels wrong, or a milestone is unclear, or you think there's a better approach, say so. You're not building this project to follow my plan. You're building it to become an engineer who can make good plans of your own.

Have fun

This project is interesting as you will get introduced to maanging customer documents, onboarding flows, microservices talking to each other. This is close to how we work now and building things from scractch will be fun. Enjoy!
