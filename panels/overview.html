Building services, the Moneygate way

A two-part guided build · layered architecture · EF Core · Docker · Blazor

Over the next few months, you're going to build two real backend services, inspired by what we actually do at Moneygate. You'll start with a small customer service to learn the shape of how we build things, then build a bigger document service that reuses the same shape and adds the harder pieces: file storage and service-to-service communication.

By the end you'll have a working system: customers can register and onboard, upload identity documents, and have them reviewed, all running locally with Docker, with a small Blazor UI on top.

★
This isn't a tutorial. It's a guided build. The point is not to finish, it's to understand every choice you make and be able to defend it.

Why two parts

You've worked with .NET and APIs before, so a lot of this might feel familiar at first. That's a trap. The work this time isn't how to write a controller, it's why we structure things the way we do.

We build the customer service first on purpose. It's plain CRUD: create a customer, update them, list them, move them through onboarding states. No file uploads, no second service to talk to. That lets you learn the skeleton (the layers, the database, validation, Docker) on an easy problem. Then the document service is the same skeleton, but bigger, where you add blob storage, file streaming, and the cross-service call. You're never fighting the architecture and the hard parts at the same time.

The layered structure we use

Both services use the same set of layers. Each layer is a separate project with a clear job. The golden rule: dependencies point inward, toward the Domain. The Domain depends on nothing.

  • Domain — entities, enums, business rules. The heart of the service. No dependencies on anything else.
  • Application — the contracts. Interfaces like ICustomerStore, the DTOs, and the validation rules. It says what the service needs, not how it's done.
  • Service — the use cases. Classes like CustomerService that orchestrate the business logic. They depend on the Application's interfaces, never on a concrete database.
  • Store — the data layer. EF Core, the DbContext, the concrete implementation of the store interfaces. This is the only layer that knows about the database.
  • Api — controllers and Program.cs. The composition root that wires everything together through dependency injection.
  • Common — small shared helpers used across layers, like a ServiceResult type. Keep it minimal.
⌥
Where do "services" go? A class like CustomerService lives in the Service layer. It depends on an interface (e.g. ICustomerStore) that's defined in Application and implemented in Store. That one idea, define the contract in one place, implement it in another, wire them in the Api, is the most important thing you'll learn. It's what makes the whole service testable.

We are deliberately keeping this simple. No MediatR, no message queues, no CQRS frameworks. Just plain classes wired through dependency injection. Once you understand the pattern by hand, adding those tools later is easy. Learning them at the same time as everything else is not.

What you're learning

Across both parts, the technical surface you'll cover:

  • C# in depth: records, generics, LINQ, async/await, pattern matching, enums, value types
  • .NET ecosystem: the difference between C# the language and .NET the platform, the SDK, the runtime
  • Layered architecture: Domain, Application, Service, Store, Api and why each exists
  • EF Core: DbContext, migrations, enum mapping, async queries, change tracking
  • Validation: FluentValidation, keeping rules out of controllers
  • REST API design: verbs, status codes, the request pipeline, fake auth via headers
  • Docker and Docker Compose: containerizing a .NET app plus MSSQL, running the whole thing locally
  • Blazor Server: a small UI that calls your API
  • Azure Blob Storage (Part 2): via Azurite, the local emulator
  • Service-to-service communication (Part 2): IHttpClientFactory, typed clients, why services own their own data
  • Testing: unit tests with xUnit, mocking, integration tests

How we'll work together

Daily async update

Each working day, post a short message in Teams at end of day. Three lines is enough: what you did, what you're stuck on, what's next. This keeps me in the loop without needing a meeting.

Real conversation at each step

When you finish a step (or near the end of one), we'll have a proper 30-45 minute call. We'll walk through your code, you'll explain your choices, I'll push back on the ones I disagree with, and we'll set up the next step together.

Stuck threshold

If you've been stuck on something for more than an hour and you've tried at least two approaches, ping me. Stuck does not mean "I don't know the answer." Stuck means "I've tried, and I'm not sure where I'm wrong." That's exactly when to ask, and it's not a sign of weakness. The opposite. Pretending to be unstuck wastes everyone's time.

Pace is yours

The steps are sized roughly in days, but I'm not measuring you against the calendar. If a step takes longer because you're going deep, that's a good thing. The goal is comprehension, not speed. This whole thing will take more than a couple of months, and that's expected.

The journey at a glance

Each step is a "fill in the next blank." One step produces something runnable or testable, and unlocks the next.

Part 1 — Customer service (the foundation)

Build the whole skeleton on easy mode: plain customer CRUD with onboarding states. By the end it runs in Docker with a small Blazor UI.

  • Step 1 — Orientation: read, predict, scaffold the empty six-project solution
  • Step 2 — Domain: the Customer entity and OnboardingStatus enum, with tests
  • Step 3 — Application contracts: interfaces, DTOs, validation rules
  • Step 4 — Service layer: CustomerService against an in-memory fake store
  • Step 5 — Store + database: EF Core, MSSQL, enum mapping, migrations
  • Step 6 — Api layer: controllers, fake auth, first true end-to-end
  • Step 7 — Docker: containerize the Api and MSSQL, run with compose
  • Step 8 — Blazor UI: a small frontend so you see it work in a browser

Part 2 — Document service (the real thing)

Same skeleton, but bigger. Adds blob storage, file streaming, and the cross-service call where the customer service talks to the document service.

  • Steps 1-6: repeat the spine (Domain, Application, Service, Store, Api) with the document domain, plus Azure Blob via Azurite and real file upload/download
  • Step 7 — Connecting the services: the customer service calls the document service over HTTP
  • Step 8 — Full stack + stretch: everything in Docker, UI wired through, plus optional deeper features

I'd suggest reading the Part 1 steps once before starting. Get the shape, then come back to each as you reach it. The Part 2 steps are lighter for now; we'll flesh them out together once Part 1 is done and we know how you're getting on.

Things to keep in mind throughout

You will make mistakes

That is the entire point. Your first attempt won't be perfect. You'll put logic in the wrong layer. You'll refactor on day three. All of this is the work, not a failure. The intern who breezes through never learns as much as the one who hits walls and works through them.

Read more than you write, early on

When you hit something new, look at how our existing repositories do it. Most of your questions are already answered in well-known places. The skill isn't inventing solutions, it's recognizing patterns.

Defend your choices

By the end of this project, every line of code should be something you chose deliberately, even if "deliberately" means "I picked the simplest option that worked." Pretending you understand when you don't will catch up with you fast.

Push back when you disagree

If something here feels wrong, or a step is unclear, or you think there's a better approach, say so. You're not building this to follow my plan. You're building it to become an engineer who can make good plans of your own.

Have fun

This is close to how we actually work, customers, onboarding, documents, services talking to each other. Building it from scratch will be genuinely fun. Enjoy it.

Looking forward to seeing it come together.
