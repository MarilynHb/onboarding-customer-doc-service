A small Blazor UI

~3-4 days See it work End of Part 1

The reward for all that backend work: a small Blazor Server UI so you can see the service work in a browser, not just in a .http file. Keep it simple. The point is to close the loop, register a customer in a form, see them appear in a list, move one through onboarding, all hitting your real API.

What you'll build

A CustomerService.Web Blazor Server project with a few pages:

  • Customers list — a table of all customers and their onboarding status.
  • Register customer — a form that posts to POST /api/customers.
  • Customer detail — view one customer, with buttons to submit, approve, reject (admin actions guarded by the fake role).

Add the Web project to docker-compose as a third service, so the full stack (UI, Api, database) comes up together.

What you'll learn

  • Blazor Server basics: components, rendering, event handlers
  • Calling your API from C# in the browser with HttpClient
  • Why the UI talks only to the Api, never to the database directly
  • Binding a form to a request DTO and showing validation errors
  • Adding a third container to your compose stack

Suggested approach

  1. Generate the Blazor Server project and add it to the solution. Configure an HttpClient pointing at your Api's base URL.
  2. Build the customers list page first. It's the simplest: call GET /api/customers on load and render a table. Getting one read working proves the wiring.
  3. Build the register form. Bind inputs to a CreateCustomerRequest, POST it, show success or the validation errors the Api returns.
  4. Build the detail page with the submit/approve/reject buttons. For the admin actions, send the fake admin role header so they're allowed.
  5. Containerize the Web project and add it to compose. Remember the in-Docker hostname rule: the UI reaches the Api at the Api's service name, not localhost.
