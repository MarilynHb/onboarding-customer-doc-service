Documents Service · Milestone 2

The Domain layer with depth

~5 days Build it twice Encapsulation

Now we start the project. We're going to scaffold the full solution structure, then spend most of this milestone building just one thing: the Document entity. Because doing it well is harder than it sounds.

What you'll build

The complete solution structure (Documents.Domain, Documents.Application, Documents.Infrastructure, Documents.Api, plus a test project) and a fully-modeled Document entity inside Documents.Domain. The entity should be non-anemic: it should encapsulate its own state and enforce its own rules. No public setters everywhere.

A DocumentStatus value (probably an enum or a small record hierarchy depending on what you decide), and at least one value object like DocumentType (Identity, ProofOfAddress, AccountStatement) or FileMetadata (FileName, ContentType, SizeInBytes).

A handful of unit tests on the entity. No frameworks except xUnit. The tests should verify business rules: "a document in Draft status cannot be approved," "an approved document cannot be re-uploaded."

What you'll learn

  • The real difference between an entity and a value object
  • How to model state transitions in code
  • Encapsulation in C#: why public setters are usually wrong
  • Records, value types, and immutability
  • Where business rules actually live (hint: in the entity, not in a service class)
  • xUnit basics

Suggested approach

  1. Set up the solution structure. You can either use the Jason Taylor template (dotnet new ca-sln) and rename projects to Documents.*, or build it from scratch with dotnet new classlib and project references. I'd suggest the template because it gets the project references right.
  2. Sketch your domain on paper before writing code. Answer these on paper:
    • What states can a Document be in?
    • What transitions are allowed? (Draft → Submitted → UnderReview → Approved | Rejected. Rejected → Resubmitted?)
    • What can change about a document, and what's immutable?
    • What goes in the entity, and what goes in a value object?
  3. Build it twice. This is the most important exercise in the project. Write a first version that feels natural to you. Get it working with tests. Then read about DDD tactical patterns and rewrite it. Compare both versions. Keep the better one.
  4. Add the unit tests. At least 5. Each test should verify one business rule.

C# depth moment: records and value types. When you build a value object like FileMetadata, you'll hit a real C# question: should it be a class, a record, a struct, or a record struct? Each is genuinely different. Take an hour, read the Records reference, and write a short note in your notes/ folder explaining your choice. You'll come back to this kind of decision a lot.

Questions to be able to answer

  1. What's the difference between an entity and a value object in your code? Show me one of each.
  2. Why is Document.Approve() a method on the entity rather than something a DocumentService does?
  3. What states can a Document be in, and what enforces the valid transitions?
  4. If I told you "documents need a new state called Expired," where would you make that change, and what would break?
