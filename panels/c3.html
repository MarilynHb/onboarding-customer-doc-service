Customer Service · Step 3

The Store layer

~4 days EF Core + MSSQL First migration

The domain exists in memory. Now you make it persistent. By the end of this step a customer, their profile, their identifications, and their onboarding application are saved to a real MSSQL database and you can read them back. No APIs yet, no service layer, no validators. Just the domain meeting the database.

This is where things stop feeling abstract. You will run a migration, look at the tables it created, insert rows from code, and query them. Once data is real, the rest of the architecture has something concrete to wrap around.

What you'll build

Inside CustomerService.Store:

  • CustomerDbContext with a DbSet for each entity: Customers, IndividualProfiles, CompanyProfiles, CustomerIdentifications, OnboardingApplications.
  • OnModelCreating overrides where you need them: the one-to-one between Customer and each profile, the one-to-many for identifications, the one-to-one with OnboardingApplication. Enums stored as strings, not ints (more on that below).
  • A connection string in appsettings.json pointing at a local MSSQL.
  • An MSSQL container running through docker-compose.yml. Azurite is not needed yet.
  • Your first migration, applied to the database.
  • A small console runner or a test that creates a Customer with a profile and an identification, saves it, and reads it back. Proof the wiring works.

What you'll learn

  • EF Core: DbContext, DbSet, OnModelCreating, change tracking, SaveChangesAsync
  • How navigation properties become foreign keys in the database
  • Why enums stored as strings are safer than enums stored as ints
  • EF Core migrations: how they are generated, how they are applied, what they actually do
  • Running MSSQL in Docker, and what to use on Apple Silicon
  • Connection strings, where they live, and what each piece means

Suggested approach

  1. Add the EF Core packages to CustomerService.Store: Microsoft.EntityFrameworkCore, Microsoft.EntityFrameworkCore.SqlServer, Microsoft.EntityFrameworkCore.Design.
  2. Write CustomerDbContext. Start with the DbSet properties only. Build it.
  3. Add docker-compose.yml at the solution root with an MSSQL service. Use mcr.microsoft.com/mssql/server:2022-latest on Windows or Intel Mac. On Apple Silicon, use mcr.microsoft.com/azure-sql-edge instead — the full SQL Server image will not run there.
  4. Run docker compose up. Verify you can connect with Azure Data Studio or your tool of choice.
  5. Put the connection string in appsettings.json on whichever startup project will run migrations (likely CustomerService.Api if it exists, or a temporary console project).
  6. In OnModelCreating, configure what EF cannot guess: the one-to-one relationships (Customer to IndividualProfile, etc.), and the enum-to-string conversion using .HasConversion<string>() on each enum property.
  7. Run your first migration: dotnet ef migrations add InitialCreate --project CustomerService.Store --startup-project CustomerService.Api. Open the generated migration file and read it. Every line corresponds to something you wrote.
  8. Apply it: dotnet ef database update. Open the database and look at the tables.
  9. Write a small piece of code that constructs a Pending Customer with an IndividualProfile and one CustomerIdentification, attaches a Draft OnboardingApplication, calls SaveChangesAsync, then queries it back with Include for the navigation properties. Confirm everything round-trips.
◆
C# depth moment: enums in the database. By default, EF Core stores an enum as its integer value. That means if you reorder OnboardingApplicationStatus later, the database becomes a mystery. InReview might suddenly mean Rejected if the order changed. Store enums as strings instead, with property.HasConversion<string>(). The column gets bigger; the system gets safer.
⌥
Design note: annotations now, Fluent API later. Your entities have [Required] and [MaxLength] annotations from the Domain step. EF Core picks those up automatically, so the database constraints will reflect them. That's fine for now. Later, a real argument can be made for moving these to Fluent API configuration in the Store layer, so the Domain stays free of persistence concerns. We will revisit this when the lesson lands better. For now: annotations stay where they are.
