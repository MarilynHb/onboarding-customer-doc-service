Customer Service · Step 3

The Store layer

~4 days EF Core + MSSQL First migration

The domain exists in memory. Now you make it persistent. By the end of this step a customer, their profile, their identifications, and their onboarding application are saved to a real MSSQL database and you can read them back. No APIs yet, no service layer, no validators. Just the domain meeting the database.

This is where things stop feeling abstract. You will run a migration, look at the tables it created, insert rows from code, and query them. Once data is real, the rest of the architecture has something concrete to wrap around.

What you'll build

Inside CustomerService.Store:

  • CustomerManagementDbContext with a DbSet for each entity: Customers, IndividualProfiles, CorporateProfiles, CustomerIdentifications, OnboardingApplications.
  • OnModelCreating overrides where you need them: the one-to-one between Customer and each profile, the one-to-many for identifications, the one-to-one with OnboardingApplication.
  • A connection string in appsettings.json in a new CustomerService.Api project, pointing at your local MSSQL container.
  • Your first migration, applied to the database.

What you'll learn

  • EF Core: DbContext, DbSet, OnModelCreating, change tracking, SaveChangesAsync
  • How navigation properties become foreign keys in the database
  • EF Core migrations: how they are generated, how they are applied, what they actually do
  • Connection strings, where they live, and what each piece means
  • Docker basics: images vs containers, and running SQL Server locally without a full install

Suggested approach

◆
New tool: Docker. You are about to run SQL Server without installing SQL Server. Docker lets you run software inside isolated containers — lightweight, self-contained environments that bundle everything a program needs to run. Two concepts to internalize before you start:
  • An image is a read-only blueprint (a packaged snapshot of the server and all its dependencies). You pull it once from a registry and it lives on your machine.
  • A container is a running instance of an image. You can start it, stop it, and throw it away. Nothing leaks onto your host machine outside the container.
For local development this is far cleaner than a full SQL Server install: one command starts the database, another stops it, and your machine stays clean.
  1. Pull the SQL Server image. Open a terminal and run: 
    docker pull mcr.microsoft.com/mssql/server:2022-latest
    This downloads the official Microsoft SQL Server 2022 image from the Microsoft Container Registry. Pulling means downloading — Docker fetches every layer of the image and stores it locally. This only needs to happen once. After that the image is cached and future containers start without re-downloading anything.
  2. Start a SQL Server container. On Windows or Intel Mac, run: 
    docker run -e "ACCEPT_EULA=Y" ^
  -e "MSSQL_SA_PASSWORD=YourPassword123!" ^
  -p 1433:1433 ^
  --name sqlserver ^
  -d mcr.microsoft.com/mssql/server:2022-latest
    On Apple Silicon (M1/M2/M3), use mcr.microsoft.com/azure-sql-edge instead — the full SQL Server image does not run on ARM. What each flag does:
    • -e "ACCEPT_EULA=Y" — SQL Server requires you to accept the license agreement. Without this the container exits immediately.
    • -e "MSSQL_SA_PASSWORD=..." — Sets the password for the sa (system administrator) login. SQL Server enforces complexity rules: at least 8 characters, mixed case, at least one digit, at least one symbol.
    • -p 1433:1433 — Maps port 1433 on your machine to port 1433 inside the container. 1433 is SQL Server's default port. The format is host-port:container-port.
    • --name sqlserver — Gives the container a name so you can reference it with docker stop sqlserver, docker start sqlserver, etc., instead of using its auto-generated ID.
    • -d — Detached mode. The container runs in the background and you get your terminal back immediately.
  3. Verify the container is running. 
    docker ps
    You should see a row for sqlserver with a status of Up X seconds. If it does not appear, run docker ps -a to list stopped containers too, then inspect the logs: 
    docker logs sqlserver
    The most common cause of failure is a password that doesn't meet the complexity rules — try a stronger one and re-run.
  4. Connect with SSMS or Azure Data Studio. Use these connection details:
    • Server name: localhost,1433
    • Authentication: SQL Server Authentication
    • Login: sa
    • Password: YourPassword123!
    • Encryption: Set to Optional (or disable)
    • Trust server certificate: Check this. Docker containers use self-signed certificates, so you need to explicitly trust them or the connection will be refused.
    • Connection timeout: The default 15 seconds is fine.
    Once connected you will see the server in Object Explorer with only the system databases. No application database yet — that appears after you run the migration.
  5. Create CustomerService.Api. Add a new ASP.NET Core Web API project to the solution. It has two roles in this step: it is the startup project that the EF CLI needs to bootstrap your application and discover the DbContext, and it is where the connection string lives in appsettings.json. It does not need to expose any endpoints yet.
  6. Add the EF Core NuGet packages — in the right projects. Not every package belongs everywhere. Here is what goes where and why:
    Microsoft.EntityFrameworkCore
    Add to: CustomerService.Store
    Core EF library — DbContext, DbSet, LINQ-to-SQL translation. The Store project is the only one that should have a direct dependency on EF.
    Microsoft.EntityFrameworkCore.SqlServer
    Add to: CustomerService.Store
    The SQL Server database provider. EF Core is database-agnostic by design; this package is what teaches it how to talk to MSSQL specifically.
    Microsoft.EntityFrameworkCore.Design
    Add to: CustomerService.Api
    Design-time tooling required by the dotnet ef CLI. It must live in the startup project because that is the project dotnet ef boots to find your DbContext and read configuration. It is not needed at runtime.
    To add a package from the terminal (run from the solution root): 
    dotnet add CustomerService.Store package Microsoft.EntityFrameworkCore
dotnet add CustomerService.Store package Microsoft.EntityFrameworkCore.SqlServer
dotnet add CustomerService.Api package Microsoft.EntityFrameworkCore.Design
    You also need the EF global CLI tool if you haven't installed it before: 
    dotnet tool install --global dotnet-ef
  7. Write CustomerManagementDbContext in CustomerService.Store. Start with the DbSet properties only. Build it and make sure it compiles before moving on.
  8. Add the connection string to appsettings.json in CustomerService.Api: 
    "ConnectionStrings": {
  "DefaultConnection": "Server=localhost,1433;Database=CustomerManagement;User Id=sa;Password=YourPassword123!;TrustServerCertificate=True;"
}
    Then register the DbContext in Program.cs: 
    builder.Services.AddDbContext<CustomerManagementDbContext>(options =>
    options.UseSqlServer(builder.Configuration.GetConnectionString("DefaultConnection")));
    TrustServerCertificate=True is required because the Docker container uses a self-signed certificate — the same reason you checked that box in SSMS.
  9. Configure relationships in OnModelCreating. Add explicit configuration for the one-to-one relationships (Customer to IndividualProfile, Customer to CorporateProfile, Customer to OnboardingApplication). EF Core does not infer one-to-one by convention — it needs you to spell them out.
  10. Run your first migration. From the solution root (not inside a project folder): 
    dotnet ef migrations add InitialCreate --project CustomerService.Store --startup-project CustomerService.Api
    • --project CustomerService.Store — tells EF where the DbContext lives and where to write the migration files.
    • --startup-project CustomerService.Api — the project EF boots to read your DI registration and connection string. This is why Design must be added there.
    Open the generated file under CustomerService.Store/Migrations/ and read it. Every CreateTable call corresponds to an entity you wrote, and every column corresponds to a property.
  11. Apply the migration to the database. 
    dotnet ef database update --project CustomerService.Store --startup-project CustomerService.Api
    Then go back to SSMS or Azure Data Studio and refresh. You should see the CustomerManagement database with all your tables.
◆
C# depth moment: one-to-one relationships in EF Core. The Customer-Profile and Customer-Application relationships are one-to-one. This is less common than one-to-many, and EF Core doesn't have a simple convention for it. You need to configure it explicitly in OnModelCreating. Follow the docs carefully to get it right.
