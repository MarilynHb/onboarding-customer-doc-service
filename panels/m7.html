Bringing it together · Milestone 7

Full Docker Compose orchestration

~3 days Calm week Polish

By now your docker-compose has grown organically. This milestone is about cleaning it up, making it production-shaped, and getting the developer experience right. Should be the calmest week of the project.

What you'll build

A polished docker-compose.yml that brings up the entire stack with one command (docker compose up):

  • Azurite (already there)
  • MSSQL (already there, two databases inside one server)
  • Documents.Api (already there)
  • CustomerOnboarding.Api (already there)
  • An init-db service or script that creates the two databases and runs migrations on startup
  • Proper service dependencies (depends_on) and health checks where it matters
  • A consolidated .env file for ports, passwords, connection strings
  • A README.md at the project root explaining how to start, stop, debug, and reset the stack

By the end, anyone (you, me, a teammate) should be able to clone the repo and have everything running in under 5 minutes.

What you'll learn

  • Docker Compose features: volumes, networks, environment variables, depends_on, healthchecks
  • Container naming and service discovery within a compose network
  • Differences between development and production Docker setups
  • Persistent vs ephemeral storage in containers
  • How to debug a containerized .NET app

Suggested approach

  1. Audit your existing docker-compose.yml. Is anything hardcoded? Move it to .env.
  2. Add health checks. MSSQL takes ~30 seconds to be ready. Documents.Api shouldn't start until MSSQL is healthy.
  3. Add a startup script that creates the two databases (DocumentsDb, CustomerOnboardingDb) if they don't exist, and runs migrations.
  4. Set up volumes for persistence. Azurite data and MSSQL data should survive a docker compose down. They should not survive a docker compose down -v.
  5. Write the README. Include: prerequisites, how to start, how to stop, how to reset, how to look at logs, common gotchas.
