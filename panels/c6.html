Customer Service · Step 6

Testing with xUnit

~3 days xUnit + Moq First test suite

You have a domain, a database, an Application layer with handlers and validators, and a Service layer with a real onboarding flow. That is a lot of behaviour to keep working as you change things. This is the right moment to add tests.

Testing is its own step on purpose. It is not a side concern to tack onto something else. It is a skill that pays off across everything you have built, and it lands better when you have real things to test rather than abstract ones.

What you'll build

Inside CustomerService.Tests:

  • Validator tests. For each validator from Step 4, a small set of tests that confirm what is valid and what is not. CreateCustomerRequestValidator_ShouldFail_WhenEmailIsInvalid, etc. These are the easiest tests in your project: pure input, pure output, no mocks.
  • Service tests for CustomerOnboardingService. These need ICustomerStore to be a stand-in (a Moq mock or a hand-written fake). Test the onboarding flow: a valid request produces a customer in Pending with the right profile. Test the approval flow: approving an InReview application flips the customer to Active; approving a Draft application returns a failure.
  • Handler tests for at least one handler. Use Moq to mock the service. Confirm the handler returns the right response on success, and translates a service failure into the right kind of response.
  • An xUnit project setup that uses the right packages: xunit, xunit.runner.visualstudio, Moq, FluentAssertions (optional but recommended for readable assertions).

You do not need to test the Store layer with EF Core yet. That belongs in integration tests, which come later. For now, mock ICustomerStore wherever the service or a handler needs it.

What you'll learn

  • xUnit basics: [Fact], [Theory] with [InlineData], test naming conventions, the AAA pattern (Arrange, Act, Assert)
  • How to mock an interface with Moq: new Mock<ICustomerStore>(), Setup, Returns, Verify
  • How to test a validator with FluentValidation's testing helpers
  • When a fake (hand-written) beats a mock (Moq), and vice versa
  • Why we don't test private methods, only behaviour
  • How a good test reads: someone unfamiliar with your code should be able to read the test name and the body and understand what is being checked

Suggested approach

  1. Make sure CustomerService.Tests already references the projects it needs (Domain, Application, Service, Common). Add Moq and FluentAssertions.
  2. Start with one validator test. The simplest case: a fully valid CreateCustomerRequest should produce no errors. Then add a few invalid cases. Use FluentValidation's TestValidate helper.
  3. Write one service test next. Build a valid request. Set up a mock ICustomerStore. Call CreateAsync. Assert the result is a success and the customer was saved with Pending status. Use mockStore.Verify(...) to confirm AddAsync was called with the right shape.
  4. Write the harder service test: ApproveAsync on a Draft application returns a failure. Set up the mock to return a Draft application. Call ApproveAsync. Assert the result is a failure with a clear reason. Assert no save happened.
  5. Write one handler test. Mock the service this time. Confirm the handler unwraps a successful result into a response, and translates a failure into the right kind of failure response.
  6. Run the tests. Watch them pass. Then deliberately break something small in the service code and watch a test fail. That is the loop you are building.
⌥
Design note: test names should read like sentences. A bad test name is Test1. A merely-fine one is TestCreateCustomer. A good one is CreateAsync_ShouldReturnFailure_WhenEmailIsAlreadyTaken. The third one tells you what is being tested, what the expected behaviour is, and the condition that triggers it. The test list becomes a readable spec of how your code behaves.
◆
C# depth moment: [Theory] over [Fact] when there is a pattern. If you find yourself copy-pasting the same test with a different input, that is a [Theory]. [InlineData("not-an-email")], [InlineData("")], [InlineData(null)] on one method runs the same assertions across three inputs. Less code, more coverage, and the failures tell you exactly which input broke.

Questions to be able to answer

  1. What's the difference between a [Fact] and a [Theory]?
  2. You're testing CustomerOnboardingService. Why mock ICustomerStore instead of using a real DbContext?
  3. One of your tests has three nearly-identical copies for different inputs. What's the better shape?
  4. Walk me through one of your test names. What does it tell me before I read the body?
  5. If a test passes when it shouldn't, what's wrong: the test, the code, or both? How would you find out?
