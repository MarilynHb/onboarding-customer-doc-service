Part 1 · Customer service · Step 6

The Api layer

~3-4 days Controllers First true end-to-end

Now you expose everything over HTTP. You'll build controllers in CustomerService.Api, wire all the layers together through dependency injection in Program.cs, add a simple fake auth, and test it with a .http file. By the end you have a real REST API you can call.

What you'll build

  • A CustomersController with endpoints:
    • POST /api/customers (register)
    • GET /api/customers (list)
    • GET /api/customers/{id} (one)
    • PUT /api/customers/{id} (update)
    • POST /api/customers/{id}/submit
    • POST /api/customers/{id}/approve (admin)
    • POST /api/customers/{id}/reject (admin)
  • DI wiring in Program.cs: register CustomerService, EfCustomerStore, the DbContext, the validators.
  • A small fake auth middleware that reads X-User-Id and X-User-Role headers and attaches them as claims, so admin endpoints can use [Authorize(Roles = "Admin")].
  • A .http file with sample requests for every endpoint, including the fake auth headers.

What you'll learn

  • ASP.NET Core controllers, routing, model binding, action results
  • How a controller stays thin: it takes the request, calls the service, returns the result
  • The composition root: where all the layers actually get connected
  • Middleware and the request pipeline
  • Mapping a ServiceResult to the right HTTP status code (200, 201, 400, 404)

Suggested approach

  1. Wire up DI in Program.cs first. Register the DbContext with the connection string, register ICustomerStore → EfCustomerStore, register CustomerService, register the validators. Get it building.
  2. Write CustomersController. Keep the controllers thin: each action calls a service method and translates the ServiceResult into an HTTP response. A failed validation becomes a 400, a not-found becomes a 404, a successful create becomes a 201.
  3. Add the fake auth middleware. Read the two headers, build a ClaimsPrincipal, assign it to HttpContext.User. Put it in the pipeline before UseAuthorization().
  4. Put [Authorize(Roles = "Admin")] on approve and reject. Test that a non-admin role gets a 403.
  5. Write the .http file and walk through every endpoint by hand. Register a customer, list them, approve one as admin, try to approve one as a non-admin and watch it get rejected.
⌥
Keep controllers boring. A common mistake is to put business logic in the controller. Don't. The controller's only job is to translate between HTTP and your service. If you find yourself writing an if about a business rule in the controller, that rule belongs in the service or the entity. Write a note: what's the most logic a controller action should ever contain?
◆
C# depth moment: delegates and the middleware pipeline. Middleware is literally a chain of functions, each taking the request and a "next" delegate. This is Func and Action made concrete. Write your fake auth middleware twice, once as an inline app.Use(...) lambda and once as a proper class, and compare them in a note.
