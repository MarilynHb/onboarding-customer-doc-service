Part 1 · Customer service · Step 1

Orientation and scaffolding

~2-3 days Read + predict Empty skeleton

Before writing any real logic, you'll read a little, make some predictions, and set up the empty solution with all six projects wired together. Nothing runs yet, but by the end the skeleton exists and you understand what each piece is for. This is the highest-leverage time in the whole project, so don't rush it.

What you'll build

An empty but correctly-wired solution called CustomerService, with six projects:

CustomerService.Domain
CustomerService.Application
CustomerService.Service
CustomerService.Store
CustomerService.Api
CustomerService.Common
CustomerService.Tests

Plus a short written reflection in notes/orientation.md: before you create the projects, predict what each layer will hold and which layers can reference which. After scaffolding, compare your predictions to what makes sense.

What you'll learn

  • The actual difference between C# (the language) and .NET (the platform). The most common confusion for new .NET developers.
  • What a .sln file is, what a .csproj file is, what bin and obj are, what Program.cs does.
  • What "layered architecture" means and why dependencies point inward toward the Domain.
  • How project references express the dependency rules.

Suggested approach

  1. Read the resources below. Take notes as you go.
  2. Before creating anything, write your predictions in notes/orientation.md. For each of the six layers, one or two sentences: what goes in it, and which other layers it's allowed to reference.
  3. Create the solution and the projects from the CLI:
dotnet new sln -n CustomerService

dotnet new classlib -n CustomerService.Domain
dotnet new classlib -n CustomerService.Application
dotnet new classlib -n CustomerService.Service
dotnet new classlib -n CustomerService.Store
dotnet new classlib -n CustomerService.Common
dotnet new webapi   -n CustomerService.Api
dotnet new xunit    -n CustomerService.Tests

dotnet sln add **/*.csproj
  1. Wire up the project references so dependencies point inward. The rough shape:
    • Application references Domain (and Common)
    • Service references Application and Domain
    • Store references Application and Domain
    • Api references Service, Store, Application, Domain
    • Domain references nothing
    Example: dotnet add CustomerService.Service reference CustomerService.Application
  2. Run dotnet build. It should compile with empty projects. Open every .csproj and read it. Open the Api's Program.cs and read what the template generated.
  3. Update notes/orientation.md with how your predictions compared to reality.
◆
The one rule to internalize now: try to add a reference from Domain to Store. Notice you can do it technically, but you shouldn't. The discipline of "Domain depends on nothing" is something you enforce, not something the compiler enforces for you. Write a sentence in your notes about why that rule matters.

  1. What's the difference between C# and .NET? Are C# 12 and .NET 8 the same thing?
  2. Why does Domain reference nothing? What would go wrong if it referenced Store?
  3. What's Program.cs doing in the Api project?
  4. If you wanted to swap the database later, which layers would change and which wouldn't?
