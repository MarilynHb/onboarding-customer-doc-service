Customer Service · Step 1

Orientation and scaffolding

~2 days No code yet Read + predict

You will not build features yet. Before any of that, you need a feel for the shape of the project and a working skeleton to fill in. This step is short on output and long on understanding. The goal is that by the end of it, you know what each layer of the solution is for and why, and you have an empty solution ready to receive the Domain layer in Step 2.

What you'll build

  • An empty solution called CustomerService with these projects:
    • CustomerService.Domain — entities, enums, base classes
    • CustomerService.Application — interfaces, DTOs, validators, handlers
    • CustomerService.Service — orchestration of flows that touch multiple entities or have rules
    • CustomerService.Store — EF Core, the DbContext, the database
    • CustomerService.Api — controllers, the entry point, DI wiring
    • CustomerService.Common — small shared types like Result
    • CustomerService.Tests — xUnit project for everything you will test
  • Project references set up so each layer can only see the layers it is allowed to see. (See "Dependency rules" below.)
  • A short note in a file called NOTES.md at the root of the repo, where you write your predictions about what each layer will hold and why. You'll come back to this and check yourself.

What you'll learn

  • The difference between C# (the language) and .NET (the platform). These are not the same thing and most people new to .NET conflate them.
  • What lives in a .sln file, a .csproj file, and where bin and obj come from
  • How project references work in .NET, and why they matter for keeping a codebase honest
  • What "layered architecture" means in plain terms, before any framework adds vocabulary on top
  • What Program.cs does in modern ASP.NET (the top-level statements style)

Suggested approach

  1. Read the resources below before you touch a keyboard. Take notes.
  2. Open NOTES.md. Without looking up anything yet, write one or two sentences for each layer above: what you think goes in it, and why. Even if you are unsure. Especially if you are unsure.
  3. Create the solution and the seven projects using the .NET CLI: 
    dotnet new sln -n CustomerService
dotnet new classlib -n CustomerService.Domain
dotnet new classlib -n CustomerService.Application
dotnet new classlib -n CustomerService.Service
dotnet new classlib -n CustomerService.Store
dotnet new classlib -n CustomerService.Common
dotnet new webapi   -n CustomerService.Api
dotnet new xunit    -n CustomerService.Tests
dotnet sln add **/*.csproj
  4. Set up project references following the dependency rules below. From the solution folder: 
    dotnet add CustomerService.Application reference CustomerService.Domain CustomerService.Common
dotnet add CustomerService.Service     reference CustomerService.Application CustomerService.Domain CustomerService.Common
dotnet add CustomerService.Store       reference CustomerService.Application CustomerService.Domain CustomerService.Common
dotnet add CustomerService.Api         reference CustomerService.Service CustomerService.Store CustomerService.Application CustomerService.Domain CustomerService.Common
dotnet add CustomerService.Tests       reference CustomerService.Service CustomerService.Application CustomerService.Domain CustomerService.Common
  5. Open the Api's Program.cs in your editor. Read what the template generated. Notice the top-level statements. Notice the lack of a Main method. Confirm the project builds with dotnet build.
  6. Go back to NOTES.md. Update your predictions if reading any of the resources changed your mind. You will return to this note after Step 2 and again after Step 4.

Dependency rules

The whole point of separate projects is to limit what can see what. The rules:

  • Domain sees nothing. It is the centre.
  • Common sees nothing or only Domain. It holds small shared types.
  • Application sees Domain and Common.
  • Service sees Application, Domain, Common.
  • Store sees Application, Domain, Common.
  • Api sees everything. It is the composition root: where things get wired up.
  • Tests sees everything it needs to test.

If a layer needs to reach into another layer it shouldn't see, that is a design smell. The compiler will catch it because the reference won't exist.

⌥
Why so many projects. One project compiles faster. Seven projects make it impossible to accidentally mix concerns. When Domain physically cannot reference Store, you cannot put a database call in your entity. The structure enforces the discipline.
◆
C# depth moment: C# vs .NET. C# is a language. .NET is a platform that includes a runtime, a base class library, and a set of frameworks (ASP.NET Core, EF Core). You write C# 12 and target .NET 8, for example. They version independently. Read the "tour of C#" and the ".NET overview" links below back to back; the distinction will become clear.
