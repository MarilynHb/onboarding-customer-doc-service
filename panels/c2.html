Part 1 · Customer service · Step 2

The Domain layer

~3-4 days Entity + enum First tests

Now you build the first real thing: the Customer entity and the OnboardingStatus enum, living in CustomerService.Domain. This layer has no dependencies. It's just C# describing what a customer is and what rules it must always obey.

What you'll build

  • An OnboardingStatus enum: Registered, InProgress, Submitted, Approved, Rejected.
  • A Customer entity with: Id, FirstName, LastName, Email, DateOfBirth, OnboardingStatus, CreatedAt.
  • Business rules enforced inside the entity, not in a service. For example: a customer starts as Registered; you can't jump straight from Registered to Approved; email can't be empty.
  • A handful of unit tests in CustomerService.Tests that prove those rules. No frameworks beyond xUnit.

What you'll learn

  • How to model an entity that protects its own state (no public setters everywhere)
  • How to represent a small state machine with an enum and guard methods
  • Where business rules live (in the entity, not scattered around)
  • xUnit basics: [Fact], [Theory], arrange-act-assert
  • How enums work in C# and how they'll later map to the database

Suggested approach

  1. Sketch the state machine on paper first. What states exist? Which transitions are allowed? Registered → InProgress → Submitted → Approved | Rejected. Can a rejected customer try again?
  2. Write the OnboardingStatus enum.
  3. Write the Customer entity. Make the constructor enforce the starting rules (status begins at Registered, email required). Add methods like SubmitForReview(), Approve(), Reject() that enforce valid transitions and throw if called at the wrong time.
  4. Resist the urge to add public setters on everything. Properties should be private set or set only through methods. The entity guards itself.
  5. Write at least 5 unit tests. Each one verifies a single rule: "new customer starts Registered", "approving a Registered customer throws", "submitting moves to Submitted", and so on.
◆
C# depth moment: enums and value types. An enum is a value type backed by an integer. That matters when it hits the database later (do you store the number or the name?). Take 30 minutes to read how enums work, and write a note: what's the risk of storing the integer value, and what happens if you reorder the enum members later?

Questions to be able to answer

  1. Why is Customer.Approve() a method on the entity rather than something a service does?
  2. What states can a customer be in, and what enforces the valid transitions?
  3. Why avoid public setters? What could go wrong if every property were freely settable?
  4. If we needed a new state like OnHold, where would you add it and what would you have to update?
