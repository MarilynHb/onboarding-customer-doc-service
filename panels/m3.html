Documents Service · Milestone 3

The Application layer

~5 days Use cases No database yet

Now we add the use-case layer. The pattern: one use case = one handler. Each handler is its own small class with one job. No database yet. We're going to fake data with an in-memory list, because the whole point of this layer is that it doesn't care where data comes from.

What you'll build

The Documents.Application project. Inside it:

  • Use cases as plain handler classes. One class per use case, each with a single Handle method:
    • UploadDocumentHandler
    • ListMyDocumentsHandler
    • GetDocumentByIdHandler
    • ResubmitDocumentHandler
  • Request and response DTOs (these are not your domain entities)
  • Validation rules using FluentValidation
  • An interface IDocumentRepository defined here (implementation in Infrastructure later)
  • An in-memory fake implementation of IDocumentRepository for now

A few unit tests on a handler. Mock the repository, verify the handler does the right thing.

What you'll learn

  • The use-case / handler pattern (sometimes called CQRS-lite)
  • How to structure an Application layer without a mediator library, using plain classes and DI
  • DTOs and why they're separate from entities
  • Validation with FluentValidation
  • The dependency inversion principle: Application defines what it needs, Infrastructure provides it
  • Why interfaces are declared in Application, not Infrastructure

Suggested approach

  1. Decide on your handler shape. The simplest pattern: each handler is a class with one public method, e.g. Task<ListMyDocumentsResponse> Handle(ListMyDocumentsRequest request, CancellationToken ct). Each handler depends on whatever it needs through constructor injection.
  2. Build ListMyDocumentsHandler first. It's the simplest. Three classes: the request DTO, the response DTO, and the handler itself.
  3. Build UploadDocumentHandler second. Don't worry about actual file handling yet, just take a fake byte[] as input.
  4. Register your handlers in DI in Program.cs. They're regular services. Scoped lifetime is the safe default.
  5. Add FluentValidation. Validate that filenames are not empty, that document types are valid, etc.
  6. Write tests. Inject a mock IDocumentRepository. Verify the handler calls the right methods.
⌥
A note on mediator libraries. You might see MediatR mentioned in tutorials or in the Jason Taylor template. We're deliberately not using it. A mediator hides the wiring between your controllers and handlers, which is great in production but bad for learning. By calling handlers directly through DI, you'll see the pattern, not the magic. Once you understand what's happening, swapping in a mediator later is a one-day exercise.

C# depth moment: generics and interfaces. As you design your handlers, you'll face a choice: should you define a generic IHandler<TRequest, TResponse> interface that all handlers implement, or just let each handler be its own class with no shared abstraction? Both are valid. Try both, see what each gains and loses. Read about generics in C# and what where T : class means. Understanding generics will save you confusion for years.
