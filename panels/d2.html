Same idea as the customer domain, but the Document entity is more interesting. It has a real state machine and a clear split between metadata (filename, type, size, status) and the file itself (which lives in blob storage, not in the entity).

What you'll build (sketch)

  • A DocumentStatus enum: Draft, Submitted, UnderReview, Approved, Rejected.
  • A DocumentType concept: Identity, ProofOfAddress, AccountStatement.
  • A Document entity holding metadata plus a reference to where the file is stored (not the bytes themselves).
  • Business rules: can't approve a Draft, an approved document is immutable, a rejected one can be resubmitted. Tests for each.

What's new compared to Part 1

  • The entity references a stored file by an opaque "storage reference", but knows nothing about how blob storage works. That's a Store concern.
  • More transitions, so more rules and more tests.
