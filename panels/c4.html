Customer Service · Step 4

The Application layer

~4 days Handlers + DTOs No MediatR

You have entities and a working database. The step after this one is an API that exposes HTTP endpoints. But between the database and those endpoints sits a layer that receives requests, validates them, and turns them into operations on your entities. That is the Application layer.

You are going to build it with plain classes and hand-wired dependency injection — no mediator library. The goal is to see the wiring clearly, not hide it.

Where this fits

The full chain from an incoming request to stored data. The parts you are building now are highlighted; the rest is already done or comes next.

For this step, your handlers call ICustomerRepository directly. In Step 5, the business logic between validation and persistence moves into a service — the handlers will shrink significantly. Building it the longer way first makes the reason for the service layer obvious.

Why not just use the DbContext in a controller?

Nothing prevents it. You could inject CustomerManagementDbContext directly into a controller and write the full flow there. It is faster on day one. Here is what it costs you:

  • You cannot test that logic without a real database — the DbContext is not something you can easily swap out in a test.
  • You cannot reuse it. If a background job or a second API endpoint needs the same logic, you copy-paste it.
  • Every controller that touches customers re-implements the same queries and validation.

A handler does not know or care whether it was called from an HTTP request, a background job, or a test. That separation is the point.

What you'll build

ICustomerRepository — in Domain

Add a Repositories folder inside CustomerService.Domain and define ICustomerRepository there. This interface describes what any code that needs customer persistence can ask for: get by id, get by customer number, add, update, delete.

Placing it in Domain rather than in a Store or Application project is a deliberate choice. Domain is the core of the architecture — it knows nothing about databases, HTTP, or any framework. By defining the interface in Domain, you ensure that everything else (Service, Application, Store) depends inward on Domain, never the other way around. Store implements the interface; it does not own it.

Then, in CustomerService.Store, create a CustomerRepository class that implements ICustomerRepository using your existing CustomerManagementDbContext.

DTOs — the shapes that cross the boundary

A DTO (Data Transfer Object) is a plain class whose only job is to carry data in or out of your application. You need the following — including nested types for the sub-objects that travel with a customer:

  • CreateCustomerRequest — the top-level create payload, carrying type, contact info, and nested sub-objects
  • IndividualProfileRequest — first name, last name, date of birth; only present when CustomerType is Individual
  • CorporateProfileRequest — company name, registration number, VAT number; only present when CustomerType is Corporate
  • IdentificationRequest — document type, number, expiry; at least one required
  • UpdateCustomerRequest — what a caller provides to update an existing customer
  • CustomerResponse — what your application returns for any read or write operation

These are not your entities. If you returned a Customer entity directly: navigation properties null unless explicitly loaded would break JSON serialisation; internal fields would be exposed to callers; and any property you add to Customer in the future would automatically appear in your public contract whether you intended that or not. A DTO gives you a stable, deliberate contract for the outside world, independent of how your domain is modelled internally.

CreateCustomerRequest carries the full graph: contact fields at the top level, exactly one profile nested inside (the type determines which), and a list of identifications. The handler constructs the equivalent entity graph from these.

Handlers — one class per use case

Each handler has one public method. It validates the request, builds entities from it, persists them via ICustomerRepository, and returns a response DTO. You need a handler for each operation:

  • CreateCustomerHandler — builds the full entity graph: Customer, the appropriate profile (IndividualProfile or CorporateProfile), all CustomerIdentification records, and the OnboardingApplication in Draft status. All of this goes into a single AddAsync call — EF Core tracks the whole graph.
  • GetCustomerHandler — retrieves a customer by id and maps it to a CustomerResponse.
  • UpdateCustomerHandler — loads the customer, applies changes to the entity (including updating profile fields if provided), persists via UpdateAsync.
  • DeleteCustomerHandler — loads the customer by id and calls DeleteAsync. EF Core cascades to related rows if your model is configured for it.

For now, handlers build the entity graph directly — constructing the customer, profile, and identifications from the request fields. This will feel slightly long. That is intentional: in Step 5, that construction logic moves to a service, and you will see exactly why the separation is useful once the handler shrinks to three lines.

Throw an exception when an operation fails (customer not found, etc.). You will introduce typed result types in Step 5.

Validators

Add a FluentValidation validator for each request DTO. An email that does not look like an email should fail before a handler ever runs. A name that is empty, a phone number that does not match a pattern — all caught at the validator. The handler should only run once the request is already known to be well-formed.

DI registration

Write an AddCustomerApplication extension method on IServiceCollection that registers handlers and validators. Write a separate AddCustomerStore method in the Store project that registers CustomerRepository as the implementation of ICustomerRepository. The Api project calls both from Program.cs.

What this looks like in code

These are illustrative skeletons showing the shape and relationships of each class — not complete implementations.

The repository interface lives in Domain and knows nothing about EF Core or SQL:

// Domain/Repositories/ICustomerRepository.cs
public interface ICustomerRepository
{
    Task<Customer?> GetByIdAsync(Guid id);
    Task<Customer?> GetByCustomerNumberAsync(string customerNumber);
    Task AddAsync(Customer customer);
    Task UpdateAsync(Customer customer);
    Task DeleteAsync(Guid id);
}

The request and response DTOs are plain classes with no behaviour:

// Application/DTOs/CreateCustomerRequest.cs
public class CreateCustomerRequest
{
    public CustomerType CustomerType { get; set; }
    public string Email { get; set; } = string.Empty;
    public string PhoneNumber { get; set; } = string.Empty;
    public IndividualProfileRequest? IndividualProfile { get; set; }
    public CorporateProfileRequest? CorporateProfile { get; set; }
    public List<IdentificationRequest> Identifications { get; set; } = [];
    // address and contact fields...
}

// Application/DTOs/CustomerResponse.cs
public class CustomerResponse
{
    public Guid Id { get; set; }
    public string CustomerNumber { get; set; } = string.Empty;
    public string Email { get; set; } = string.Empty;
    public CustomerStatus Status { get; set; }
    // other fields the caller needs...
}

The handler validates, builds the entity graph, persists, and maps to a response. Notice it is doing the construction work directly — this is what moves to the service in Step 5:

// Application/Handlers/CreateCustomerHandler.cs
public class CreateCustomerHandler
{
    private readonly ICustomerRepository _repository;
    private readonly IValidator<CreateCustomerRequest> _validator;

    public CreateCustomerHandler(
        ICustomerRepository repository,
        IValidator<CreateCustomerRequest> validator)
    {
        _repository = repository;
        _validator = validator;
    }

    public async Task<CustomerResponse> HandleAsync(CreateCustomerRequest request)
    {
        await _validator.ValidateAndThrowAsync(request);

        var customer = new Customer
        {
            Id = Guid.NewGuid(),
            Email = request.Email,
            // build profile, identifications, onboarding application...
        };

        await _repository.AddAsync(customer);
        return MapToResponse(customer);
    }

    private static CustomerResponse MapToResponse(Customer customer) => new()
    {
        Id = customer.Id,
        Email = customer.Email,
        // ...
    };
}

The validator enforces that requests are well-formed before any handler runs:

// Application/Validators/CreateCustomerRequestValidator.cs
public class CreateCustomerRequestValidator : AbstractValidator<CreateCustomerRequest>
{
    public CreateCustomerRequestValidator()
    {
        RuleFor(x => x.Email).NotEmpty().EmailAddress();
        RuleFor(x => x.PhoneNumber).NotEmpty().Matches(@"^\+?[1-9]\d{6,14}$");
        RuleFor(x => x.Identifications).NotEmpty();
        // profile rules conditional on CustomerType...
    }
}
⌥
This is not CQRS. Handler-per-use-case with request and response objects can look like CQRS (Command Query Responsibility Segregation), but it is not the same pattern. CQRS explicitly splits reads and writes at the model level: queries return a flattened read model built from a separate projection or read database; commands go through a different path entirely. Here, GetCustomerHandler and CreateCustomerHandler use the same Customer entity and the same ICustomerRepository. The naming is a stylistic choice — one class per use case — not a CQRS commitment. Full CQRS is a significant architectural investment and is not what you are building.

Where to start

The repository interface is the right starting point. Look at what operations your handlers will need to perform against the database — those become the method signatures on ICustomerRepository. Define the interface in Domain first. Then add CustomerRepository in Store and have it implement the interface using the DbContext you already have.

Once the interface and its implementation exist, think about the DTO shapes from the caller's perspective: what would someone need to provide to create a customer? What would they expect to get back? Let the use cases drive the shapes, not the database schema.

Pick one handler and get it working end-to-end — validate, build, persist, respond — before writing the others. The pattern repeats; getting the first one right makes the rest straightforward.

◆
C# depth moment: async all the way down. Your repository is async. Your handlers are async. Once you mark one method async, the change propagates up the call stack. This is by design. await does not block a thread — it returns control to the caller while the I/O finishes. That is why a server with a few threads can serve thousands of concurrent requests. If you have not read carefully about Task and what await actually does at the runtime level, now is the time.
