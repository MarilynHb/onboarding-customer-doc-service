Part 1 · Customer service · Step 4

The Service layer

~3-4 days Business logic runs In-memory fake

This is where the logic comes alive. You build CustomerService (the class) in CustomerService.Service. It uses the ICustomerStore interface from the last step, but there's still no database. You'll write a fake in-memory store so the whole thing runs and is fully testable, with nothing but C# objects in a list.

What you'll build

  • A CustomerService class with methods matching your use cases: RegisterCustomer, UpdateCustomer, GetCustomer, ListCustomers, SubmitForReview, ApproveCustomer, RejectCustomer.
  • Each method: validates input (using the validators from Step 3), maps DTO to entity (or back), calls the entity's own methods to enforce rules, and uses ICustomerStore to persist.
  • An in-memory implementation of ICustomerStore (e.g. InMemoryCustomerStore backed by a List<Customer> or a Dictionary). This can live in the Tests project or a temporary spot; it's a stand-in until Step 5.
  • Unit tests for the service, using the in-memory store or a mock.

What you'll learn

  • How a service orchestrates: validate, map, call domain methods, persist
  • How depending on an interface (not a database) keeps the service testable
  • The difference between business rules (in the entity) and orchestration (in the service)
  • How a ServiceResult-style return type communicates success vs failure without throwing for expected cases

Suggested approach

  1. Build a ServiceResult type in CustomerService.Common (success/failure plus an optional message and value). Many of our services use this pattern instead of throwing for ordinary failures like "not found" or "validation failed."
  2. Write RegisterCustomer first. Steps inside it: run the validator, if invalid return a failed ServiceResult; create the Customer entity; call store.Add; return success with a CustomerResponse.
  3. Write the in-memory store. It just holds a collection and implements ICustomerStore. This proves your service works before any database exists.
  4. Write the remaining methods. For ApproveCustomer, load the entity, call its Approve() method (which enforces the transition rule), save it.
  5. Test the service thoroughly with the in-memory store. "Registering a valid customer succeeds", "registering with a bad email fails", "approving a registered customer fails", and so on.
⌥
The big realization here: your entire service works, end to end, with zero database code. You can register, approve, and list customers, all in memory. That's not a hack, it's the proof that your layers are decoupled. In Step 5 you'll swap the fake store for a real one and the service won't change at all.
