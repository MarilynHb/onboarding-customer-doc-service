Customer Service · Step 4

The Application layer

~4 days Handlers + DTOs No MediatR

You have entities and a working database. Now you need a layer between the outside world and the database. That is the Application layer: the place where requests come in, get validated, and are turned into operations on the entities.

You are going to define an interface that the Store layer fulfils, write the request and response shapes the outside world will use, write small handler classes that do one thing each, and add validation rules. No MediatR, no mediator library. Plain classes wired by hand through dependency injection. The point is to see the wiring, not hide it.

What you'll build

Inside CustomerService.Application:

  • ICustomerStore interface. A contract describing what the Application layer needs from persistence: GetByIdAsync, GetByCustomerNumberAsync, AddAsync, UpdateAsync, DeleteAsync, plus whatever else you need. The interface lives here, in Application. The implementation already lives in Store, where the DbContext is.
  • DTOs. CreateCustomerRequest, UpdateCustomerRequest, CustomerResponse. These are not your entities. They are plain shapes for crossing the boundary, so the inside of your domain can change without breaking callers. CreateCustomerRequest should include enough to create the customer, their profile (one of two shapes depending on type), at least one identification, and the address and contact fields.
  • Handler classes. One class per use case, with one public method. CreateCustomerHandler.HandleAsync, GetCustomerHandler.HandleAsync, UpdateCustomerHandler.HandleAsync. Each handler takes its request DTO, does its work via ICustomerStore, and returns a response DTO (or some kind of result type for failures).
  • FluentValidation validators for the request DTOs. CreateCustomerRequestValidator, UpdateCustomerRequestValidator. Rules: email is an email, phone matches a pattern, names are not empty, dates are in the past where they should be.
  • DI registration. A small AddCustomerApplication extension method on IServiceCollection that registers the handlers and the validators. The Api project will call this from Program.cs.

What you'll learn

  • Why an interface defined in Application is implemented in Store: dependency inversion in practice
  • Why DTOs are not entities, and what breaks if you skip them
  • The handler pattern: small, single-purpose classes, easy to find, easy to test
  • How to validate inputs cleanly with FluentValidation
  • Manual DI wiring: which lifetimes to use, and why most things here are Scoped

Suggested approach

  1. Start with ICustomerStore. Look at the things your domain code from Step 3 actually did: load a customer, save a customer, etc. Lift those into method signatures on the interface.
  2. Move the existing CustomerStore implementation in the Store layer behind this interface (it should already be doing the right things; it just needs to implement ICustomerStore now).
  3. Write the DTOs. Start with CustomerResponse — it's the simplest. Then CreateCustomerRequest: include the profile shape conditionally on customer type, and a list of identifications.
  4. Write CreateCustomerHandler first. It takes the request, constructs the entities, hands them to ICustomerStore.AddAsync, and returns a CustomerResponse. Keep it short. If it gets long, you probably have business logic that belongs in the Service layer in Step 5.
  5. Write the rest of the handlers: GetCustomerHandler, UpdateCustomerHandler, DeleteCustomerHandler.
  6. Add the FluentValidation package. Write the validators. One validator per request DTO. Keep them strict and clear: an email that does not look like an email should fail.
  7. Write AddCustomerApplication. Inside it: register each handler as Scoped, register the validators with AddValidatorsFromAssemblyContaining.
⌥
Design note: why not just call the DbContext from a controller. Nothing stops you from injecting the DbContext into a controller and writing the whole flow there. It is faster on day one. It is slower for the rest of the project's life: you cannot test that flow without a real database, you cannot reuse the logic from a console job or another entry point, and you end up writing the same code in five controllers. The handler-and-interface separation pays off the moment you write your second consumer.
◆
C# depth moment: async all the way down. Your store is async. Your handlers are async. Your validators have an async variant. Once you mark one method async, the change propagates up the call stack. This is by design. Read about Task and what await actually does. The short version: await does not block a thread; it returns control to the caller while the I/O finishes. That is why a server with a few threads can serve thousands of requests.

  2. What's the difference between a CustomerResponse and a Customer entity? Why not just return the entity?
  3. What lifetime did you register the handlers with, and why?
  4. If a request comes in with an invalid email, where exactly does it get rejected?
  5. Could you swap MSSQL for a different database without changing any code in the Application layer? Why or why not?
