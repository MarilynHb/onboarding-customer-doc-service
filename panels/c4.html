Customer Service · Step 4

The Application layer

~4 days Handlers + DTOs No MediatR

You have entities and a working database. The step after this one is an API that exposes HTTP endpoints. But between the database and those endpoints, you need a layer that receives requests, validates them, and turns them into operations on your entities. That is the Application layer.

You are going to build it with plain classes and hand-wired dependency injection — no mediator library. The goal is to see the wiring clearly, not hide it behind abstractions.

Where this fits

Right now your chain stops at the database. After this step, the full path from an incoming request to stored data will look like this:

HTTP client — a browser, a mobile app, Postman, anything sending requests
API layer — controllers that receive HTTP requests (Step 5, not built yet)
Application layer — you are building this now
  • DTOs: the shapes that travel in and out
  • Handlers: one class per use case
  • ICustomerStore: the contract this layer requires from persistence
  • Validators: ensure requests are well-formed before any work is done
Store layer — what you built in Step 3
  • CustomerStore: will implement ICustomerStore
  • CustomerManagementDbContext: talks to the database

The API layer calls into the Application layer. The Application layer calls into the Store layer through an interface. Nothing in Application ever touches the DbContext directly.

Why not just use the DbContext in a controller?

Nothing prevents it. You could inject CustomerManagementDbContext directly into a controller and write the full flow there. It is faster on day one. Here is what it costs you:

  • You cannot test that logic without a real database — the DbContext is not something you can easily swap out in a test.
  • You cannot reuse it. If a background job or a second API endpoint needs the same logic, you copy-paste it.
  • Every controller that touches customers re-implements the same queries and validation.

A handler does not know or care whether it was called from an HTTP request, a background job, or a test. That separation is exactly the point. The payoff is visible the moment you write your second consumer.

What you'll build

Everything in this step lives inside a new CustomerService.Application project.

ICustomerStore — the interface the Application layer defines

Instead of depending directly on the Store layer, the Application layer defines an interface that describes what it needs from persistence: get a customer by ID, get one by customer number, add one, update one, delete one. The interface lives here, in Application. A CustomerStore class in the Store layer implements it using the DbContext.

This is dependency inversion in practice. Normally you might think Store is the foundation and Application builds on top of it. But here the dependency arrow runs the other way: Store depends on Application's interface, not the other way around. The consequence is that Application has no knowledge of how persistence works — it only knows what it needs. You could replace MSSQL with Postgres, or swap the DbContext for an in-memory implementation in tests, without touching a single line in Application.

⌥
On the CustomerStore class. Step 3 gave you a working DbContext and live tables. To implement ICustomerStore, you now need a CustomerStore class in the Store project that wraps the DbContext and satisfies the interface. Think about what operations your handlers will need — those become the methods. The DbContext stays inside the Store layer and never surfaces above it.

DTOs — the shapes that cross the boundary

A DTO (Data Transfer Object) is a plain class whose only job is to carry data in or out of your application. You need three:

  • CreateCustomerRequest — what a caller must provide to create a customer
  • UpdateCustomerRequest — what a caller provides to update one
  • CustomerResponse — what your application returns

These are not your entities. Consider what happens if you return a Customer entity directly from a handler: navigation properties that are null unless explicitly loaded will break JSON serialisation; internal fields that have no business being visible to callers will be exposed; and any change you make to the entity in the future — adding a field, renaming a property — immediately changes what callers receive, whether you intended that or not. A DTO gives you a stable, deliberate contract for the outside world that is entirely independent of how your domain is modelled internally.

CreateCustomerRequest should carry enough to create the customer: their type, profile details (which differ between individual and corporate), at least one identification, and their address and contact information.

Handlers — one class per use case

Each handler has one public method. It receives a request DTO, performs its work through ICustomerStore, and returns a response DTO. You need at minimum a handler for creating, getting, updating, and deleting a customer.

Keep handlers short. If a handler starts doing a lot of work — enforcing business rules, calculating derived state, coordinating across multiple entities — that is a signal that logic belonging in the Service layer (Step 5) has leaked in. The handler's job is to orchestrate, not to decide.

For now, throw an exception when an operation fails (customer not found, duplicate detected). You can revisit typed error handling later once the plumbing is in place.

Validators

Add a FluentValidation validator for each request DTO. An email that does not look like an email should fail before a handler ever runs. A name that is empty, a date that is in the future when it should be in the past, a phone number that does not match a pattern — all caught at the validator. The handler should only run when the request is already known to be well-formed.

DI registration

Write an AddCustomerApplication extension method on IServiceCollection. It registers the handlers and the validators. The Api project calls this from Program.cs, just like it already calls the DbContext registration from Step 3.

What you'll learn

  • Why the interface belongs to the caller (Application), not the implementer (Store), and what that makes possible
  • What DTOs are, why they are not entities, and what goes wrong when you skip them
  • The handler pattern: small, single-purpose classes that are straightforward to find and straightforward to test
  • Input validation with FluentValidation, and where in the chain it fits
  • Manual DI wiring: registering handlers and validators, and which lifetime to use

Where to start

Before writing any code, look at what your domain actually needs to do with the database. What operations does a handler need to be able to perform? Those become the method signatures on ICustomerStore. Start there — the interface is the contract everything else depends on, and defining it first forces you to think about what the Application layer actually needs rather than what the Store happens to offer.

Once the interface exists, think about the DTO shapes from the caller's perspective, not from the entity's perspective. What would someone sending a request need to provide? What would they expect to get back? Let the use cases drive the shapes, not the database schema.

Pick one handler and get it working end-to-end before writing the others. The path from DTO in, through the interface, to DTO out is the pattern you will repeat for every use case. Getting the first one right establishes the template.

◆
C# depth moment: async all the way down. Your store is async. Your handlers are async. Once you mark one method async, the change propagates up the call stack. This is by design. await does not block a thread — it returns control to the caller while the I/O finishes. That is why a server with a few threads can serve thousands of concurrent requests. If you have not read carefully about Task and what await actually does at the runtime level, now is the time.

Questions to be able to answer

  1. Why does ICustomerStore live in Application but its implementation live in Store? What would break if it were the other way around?
  2. What is the difference between a CustomerResponse and a Customer entity? Give a concrete example of something that would go wrong if you returned the entity directly.
  3. What lifetime did you register the handlers with, and why that one rather than Singleton or Transient?
  4. If a request arrives with an invalid email address, at what point in the chain is it rejected, and why there rather than inside the handler?
  5. Could you replace MSSQL with a different database without changing any code in the Application layer? What makes that possible?
