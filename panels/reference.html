Solution workflows (the finished system)

⌥
These two diagrams describe the completed two-service system, the state you reach at the end of Part 2. While you're building Part 1 (the customer service alone), the document service and the cross-service call don't exist yet. Treat these as the destination, not the starting point.

Upload flow

sequenceDiagram autonumber actor User as Customer participant UI as CustomerService.Web participant CO as CustomerService.Api participant CODB as Customer DB participant DOC as DocumentService.Api participant DOCDB as Document DB participant BLOB as Azurite (Blob) User->>UI: Selects file, clicks upload UI->>CO: POST document (file + user info) activate CO CO->>CODB: Load customer / application CODB-->>CO: Record + state Note over CO: Validate ownership and state CO->>DOC: POST file (stream + user info) activate DOC DOC->>BLOB: UploadAsync(stream) BLOB-->>DOC: Blob reference DOC->>DOCDB: Save document metadata DOCDB-->>DOC: Document Id DOC-->>CO: 201 Created (documentId) deactivate DOC CO->>CODB: Save document reference CODB-->>CO: OK CO-->>UI: 201 Created (documentId) deactivate CO UI-->>User: Shows success, refreshes list

The Azurite connection string (Part 2)

Memorize this, it's the same everywhere:

UseDevelopmentStorage=true

This is a shorthand the Azure SDK resolves to a full local connection string with a well-known account key. That key is not a secret, it's hardcoded by Microsoft for all Azurite users. Don't be surprised when you see it in tutorials.

EF Core migration commands

# Create a migration
dotnet ef migrations add <Name> \
  --project CustomerService.Store \
  --startup-project CustomerService.Api

# Apply migrations
dotnet ef database update \
  --project CustomerService.Store \
  --startup-project CustomerService.Api

# Generate a SQL script (for review)
dotnet ef migrations script \
  --project CustomerService.Store \
  --startup-project CustomerService.Api

Useful one-liners

# Bring up the whole stack
docker compose up

# Bring down the stack, keep data
docker compose down

# Bring down the stack and wipe data
docker compose down -v

# Tail logs from a single service
docker compose logs -f customer-api

# Restart one service
docker compose restart customer-api

# Open a shell inside a container
docker compose exec customer-api /bin/bash
