Git guide

Git & Visual Studio

A tiered reference — start at Tier 1, return to the others as you need them.

Tier 1 — Must know now

Commits = snapshots

A commit is a snapshot of your entire project at a point in time — not a diff, not a list of changes. Git stores the full state and figures out what changed. When you look at history you are looking at a chain of snapshots. Keep this mental model; everything else (branches, checkout, reset) makes more sense with it.

Branches

A branch is just a named pointer to a commit. Creating one costs almost nothing. The point: you can work on something without touching main. In Visual Studio, the current branch is shown in the bottom-left status bar. Click it to create a new branch or switch to an existing one.

💡
Always branch off main before starting new work. Never commit directly to main unless your team has explicitly said it's fine for that repo.

Staging vs. committing

In the Git Changes window (View → Git Changes), modified files appear under Changes. You stage a file by clicking the + icon next to it — this moves it to Staged Changes. Only staged files go into the next commit. This lets you commit a logical unit of work even if you have other unrelated edits sitting around.

Checkout: switching branches and detached HEAD

Checking out a branch updates your working files to match that branch's latest snapshot. Click the branch name in the status bar and pick the branch you want.

Checking out a specific commit (instead of a branch) puts you in detached HEAD state — you're no longer on any branch. Changes you make here won't be saved to any branch unless you explicitly create one. VS will warn you. If you end up here accidentally, just switch back to your branch via the status bar.

Merging and resolving conflicts

Merging brings changes from one branch into another. In VS: Git Changes → Branch → Merge (or use the Git menu). If the same lines were edited in both branches, Git can't decide automatically — you'll get a conflict. VS opens its three-panel merge editor: left is your branch, right is the incoming branch, bottom is the result. For each conflict, click Accept Current, Accept Incoming, or edit the result pane directly. Save and commit when done.

Stash: shelve and unshelve

Stashing saves your uncommitted work to a temporary shelf so you can switch branches without losing it. In Git Changes, click the three-dot menu (⋯) → Stash All. To restore it later: same menu → Pop Stash (restores and removes from the stash) or Apply Stash (restores but keeps the stash entry).

.gitignore — what not to track

bin/ and obj/ are build output — they're regenerated every build, they're large, and they differ per machine. .vs/ and *.user files store your personal IDE settings (window layout, breakpoints). None of these belong in source control. A standard .gitignore for .NET excludes them automatically. If you accidentally committed them, VS can help you remove them from tracking without deleting the files.

Reset: soft/mixed vs. hard

Reset moves the branch pointer backward to an earlier commit. The dangerous difference:

SoftYesKept, staged
Mixed (default)YesKept, unstaged
HardYesDiscarded — gone
⚠️
Hard reset discards uncommitted work permanently. Double-check before running it. If you already pushed commits and then hard-reset, you'll need a force-push to update the remote — see Tier 2 for why that's dangerous.

Pull vs. Fetch in Visual Studio

Fetch downloads the latest data from the remote but does not change your working files. It updates VS's knowledge of what the remote looks like.

Pull = Fetch + Merge. It downloads and immediately merges the remote branch into your current branch.

When VS says "your branch is X commits behind origin/main", it means the remote has X commits that you haven't merged yet. Fetch first to see what's incoming, then Pull when you're ready.

Tier 2 — Good to know soon

Reflog: the undo panic button

The reflog is a local log of everywhere HEAD has pointed. Even after a hard reset or an accidental branch delete, the commits are still there — Git just isn't pointing at them. Run git reflog in the terminal to see the list and find the SHA you want to recover, then git checkout <sha> to get back to it. You have a window of at least 30 days before Git garbage-collects unreachable commits.

Force-push danger

A regular push is rejected if the remote has commits you don't have locally (it protects you). A force-push (--force) overrides that check and rewrites the remote history. If anyone else has pulled those commits, their history now diverges from yours — they'll have to do painful reconciliation. Never force-push to a shared branch (e.g. main, develop) without explicit team agreement.

Configuration Manager

Build → Configuration Manager lets you set each project in your solution to Debug or Release independently. Debug builds include symbol files and disable optimisations — useful for stepping through code. Release builds are optimised for production. You'll normally leave everything on Debug during development and switch to Release before publishing.

Multiple startup projects

If you need to run CustomerService and DocumentService at the same time: right-click the solution → Set Startup Projects → choose Multiple startup projects and set each relevant project to Start. F5 will then launch all of them together.

Commit hygiene

Small commits are easier to review, easier to revert, and easier to understand six months later. A good commit message says why the change was made, not just what changed (the diff already shows what). Aim for one logical change per commit. A useful format: a short imperative headline (≤72 chars), then an optional blank line followed by more detail if needed.

Branch naming and workflow

Ask your team what convention they use. Common patterns: feature/short-description, fix/ticket-number-description. Consistent names make the branch list readable and help CI pipelines apply the right rules automatically.

Tier 3 — Later, if it comes up

Rebase

Rebase replays your commits on top of another branch instead of creating a merge commit. The result is a cleaner, linear history. It's worth learning once you've seen enough merges to understand what it's replacing — the mechanics are straightforward but the mental model shift takes a few rounds.

.sln / .csproj conflict resolution

Solution and project files are XML. Conflicts in them usually look worse than they are — they often happen when two people add a new project or NuGet package at the same time. Open the conflict in VS's merge editor, compare the two sides, and manually reconstruct the correct XML (usually both additions should be kept). Validate by reloading the solution.

Cherry-pick

Cherry-pick applies a single commit from one branch onto another, without merging the whole branch. Useful when a bug fix was made on a feature branch and you need it on main before the feature is ready. In VS: Git menu → Cherry-Pick (or use the terminal: git cherry-pick <sha>).

Tags

Tags are named pointers to a specific commit that don't move — unlike branches. Typically used to mark release versions (v1.0.0). Creating a tag: git tag v1.0.0. Tags aren't pushed by default; use git push origin --tags to share them.
