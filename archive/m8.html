Customer Onboarding · Milestone 8

Blazor UI and stretch goals

~6 days Open-ended Your pick

The final milestone. You build a small Blazor Server frontend that demonstrates the whole stack working end-to-end. Once that's working, you have time for stretch goals based on what interests you.

What you'll build

CustomerOnboarding.Ui, a Blazor Server app that talks to CustomerOnboarding.Api (not Documents.Api directly; only CustomerOnboarding is its consumer). Pages:

  • Register — customer enters their basic info, starts an onboarding application
  • My application — view the current application, see required documents, upload missing ones
  • Document detail — view a document's metadata and download it
  • Admin: pending applications — list of applications awaiting review (admin role)
  • Admin: review application — see all documents in an application, approve or reject

The UI uses the same fake auth pattern. No real login flow. Add the Blazor UI to docker-compose as a fifth service.

Stretch goals (pick what interests you)

  • Tests: real test coverage for handlers in both APIs, a couple of integration tests with WebApplicationFactory
  • SAS URLs: instead of streaming downloads through the API, generate short-lived SAS URLs to Azurite so the browser fetches directly
  • Audit log: every status transition gets logged to an audit table; admins can view the history
  • Structured logging: add Serilog with structured properties, log to console in dev
  • Soft delete: documents that customers delete are flagged, not removed; admins can still see them
  • Document expiry: identity documents expire after 12 months; automatic transition to Expired state
  • A small reporting endpoint: how many applications were submitted, approved, rejected per month
  • Polly for resilience: retry policies on the HTTP client between CustomerOnboarding and Documents
★
Better one stretch goal done well than four done shallowly.

What you'll learn

  • Blazor Server: components, rendering, calling APIs from C# in the browser
  • The trade-offs of Blazor Server vs WebAssembly
  • How to handle file uploads in Blazor (using InputFile)
  • Why we keep services thin and put orchestration in the consumer

Suggested approach

  1. Generate a Blazor Server project (dotnet new blazorserver). Add it to your solution.
  2. Build the "list my documents" page first. It's the simplest. Inject an HttpClient configured to point at CustomerOnboarding.Api.
  3. Build the upload page using Blazor's <InputFile> component.
  4. Build the admin review page. This shows orchestration: UI calls CustomerOnboarding, which calls Documents to get statuses, which the UI displays.
  5. Containerize and add to compose.
  6. Pick a stretch goal. Implement it well.
