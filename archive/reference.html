View Application Flow

sequenceDiagram autonumber actor User as Customer participant UI as "CustomerOnboarding.Ui (Blazor Server)" participant CO as CustomerOnboarding.Api participant CODB as CustomerOnboarding DB participant DOC as Documents.Api participant DOCDB as Documents DB User->>UI: Opens "My application" page UI->>CO: GET /applications/{id} (X-User-Id header) activate CO CO->>CODB: Load OnboardingApplication + RequiredDocuments CODB-->>CO: Application + document references Note over CO: For each reference, need current status from Documents CO->>DOC: GET /api/documents?ids=[...] (batch fetch by IDs) activate DOC DOC->>DOCDB: Query documents by IDs (filter by owner) DOCDB-->>DOC: Document metadata + statuses DOC-->>CO: 200 OK [{ id, status, fileName, ... }] deactivate DOC Note over CO: Merge: enrich references with current statuses CO-->>UI: 200 OK { application, documents[] } deactivate CO UI-->>User: Renders page with documents and their statuses

The Azurite connection string

Memorize this, it's the same everywhere:

UseDevelopmentStorage=true

This is a magic shorthand that the Azure SDK resolves to:

DefaultEndpointsProtocol=http;
AccountName=devstoreaccount1;
AccountKey=Eby8vdM02xNOcqFlqUwJPLlmEtlCDXJ1OUzFT50uSRZ6IFsuFq2UVErCz4I6tq/K1SZFPTOtr/KBHBeksoGMGw==;
BlobEndpoint=http://127.0.0.1:10000/devstoreaccount1;

That account key is not a secret. It's hardcoded by Microsoft for all Azurite users. Don't be surprised when you see it in tutorials.

EF Core migration commands

# Create a migration
dotnet ef migrations add <Name> \
  --project src/Documents/Documents.Infrastructure \
  --startup-project src/Documents/Documents.Api

# Apply migrations
dotnet ef database update \
  --project src/Documents/Documents.Infrastructure \
  --startup-project src/Documents/Documents.Api

# Generate a SQL script (for review)
dotnet ef migrations script \
  --project src/Documents/Documents.Infrastructure \
  --startup-project src/Documents/Documents.Api

Useful one-liners

# Bring up the whole stack
docker compose up

# Bring down the stack, keep data
docker compose down

# Bring down the stack and wipe data
docker compose down -v

# Tail logs from a single service
docker compose logs -f documents-api

# Restart one service
docker compose restart documents-api

# Open a shell inside a container
docker compose exec documents-api /bin/bash
