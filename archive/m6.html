Customer Onboarding · Milestone 6

The Onboarding service

~6 days Microservices Service-to-service HTTP

The big architectural moment. You've built one service. Now build a second one that consumes it. This is where microservices stops being a buzzword.

What you'll build

A complete second service: CustomerOnboarding.Api. Same architecture as Documents.Api, its own MSSQL database, but with a fundamentally different responsibility: it owns customers and onboarding applications, and it calls Documents.Api when it needs documents.

The domain

  • Customer (Id, FirstName, LastName, Email, DateOfBirth, OnboardingStatus, CreatedAt)
  • OnboardingApplication (Id, CustomerId, Status, SubmittedAt, ReviewedAt, ReviewerId)
  • RequiredDocument (a lightweight reference: ApplicationId, DocumentId, DocumentType)

Use cases

  • Register a new customer (creates the customer + a draft onboarding application)
  • Attach a document to an application (calls Documents.Api over HTTP to upload, then stores the reference)
  • Submit application for review (transitions state; business rule: all required documents must be present)
  • Get application status (calls Documents.Api to enrich the response with current document statuses)
  • Admin: list pending applications, approve/reject

The HTTP client to Documents.Api should use IHttpClientFactory with the typed client pattern. Not raw HttpClient, not named clients.

What you'll learn

  • Service-to-service HTTP communication in .NET
  • IHttpClientFactory and why it exists (port exhaustion, DNS staleness)
  • Typed clients vs named clients
  • How services "own their data": CustomerOnboarding owns customers, Documents owns documents. Neither queries the other's database.
  • Service contracts: how the two services agree on what document IDs look like, what statuses mean
  • The cost of microservices: every cross-service operation is now an HTTP call that can fail

Suggested approach

  1. Scaffold the CustomerOnboarding solution structure. Same four-project shape as Documents.
  2. Build the Customer entity and the OnboardingApplication entity. State machines as before. Tests for the business rules.
  3. Build the Application layer. Use cases first, no HTTP integration yet, in-memory fakes for both the database and the document service.
  4. Add the Infrastructure layer with a second MSSQL database (separate database, not separate server). Migrations.
  5. Now add the typed HTTP client. Define IDocumentsApiClient in Application. Implement as DocumentsApiClient in Infrastructure, registered with AddHttpClient<IDocumentsApiClient, DocumentsApiClient>(). Configure the base URL through appsettings.json.
  6. Build the controllers and the Web layer.
  7. Add CustomerOnboarding.Api to docker-compose. Inside Docker, it talks to http://documents-api:8080, not http://localhost:5001.
  8. End-to-end test: hit CustomerOnboarding.Api with a Postman request that triggers a document upload, and verify the file lands in Azurite.

C# depth moment: pattern matching and exhaustive switches. When CustomerOnboarding receives a response from Documents.Api, you'll handle different statuses (Approved, Rejected, UnderReview, etc.). This is a perfect place for modern C# pattern matching with switch expressions. Read about pattern matching in C# and rewrite at least one if/else chain as a switch expression. Notice what changes when a new case is added: the compiler tells you.

Questions to be able to answer

  1. Why does CustomerOnboarding.Api not query Documents.Api's database directly?
  2. What would happen if the Documents.Api was down when CustomerOnboarding.Api tried to upload? How does the user experience this?
  3. Why typed clients instead of just new HttpClient() each time?
  4. If you needed to add a third service tomorrow (say, NotificationsService) that also wanted to know when a document was approved, what would have to change?
  5. What's the contract between CustomerOnboarding and Documents? Where is it written down?
